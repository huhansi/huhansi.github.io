2 开始

如果你想从大体上了解Spring或者Spring Boot，那么，本章就是你需要阅读的。它会回答一些基础的“什么？”，“怎么？”和“为什么”等问题。本章介绍了Spring Boot，以及它的安装指南。然后我们会构建第一个Spring Boot应用，并讨论一些需要遵循的核心原则。

Spring Boot介绍

Spring Boot可以助你创建一个可运行的，基于Spring的，独立，生产级的应用程序。我们为Spring平台及第三方库提供开箱即用的设置，这样你就可以快速开始。多数Spring Boot应用只需要非常少的Spring配置。

你可以使用Spring Boot创建Java应用，并使用命令java -jar来启动它，或者采用传统的war包部署方式。我们也提供了一个命令行工具，运行“spring 脚本”。

我们的主要目标是：

  • 为所有的Spring开发提供一个更快，且所处可得的入门体验
  • 开箱即用。可以通过覆盖默认配置来修改程序行为
  • 提供一系列大型项目常用的非功能性特征，比如：内嵌服务器，安全，指标，健康检测，外部化配置
  • 绝对没有代码生成，也不需要XML配置

给Java开发者的安装指南

只要在对应的类路径下引入合适的spring-boot-*.jar文件，你就可以像使用其他标准Java库一样使用Spring Boot。Spring Boot 不需要集成任何工具，你可以使用任何的文本编辑器或者集成开发环境。同时，Spring Boot也没有什么独特之处，你可以像对待其他Java程序那样运行它，调试它。

3.1 构建系统

我们强烈建议你选择一个支持依赖管理，可以从中央仓库消费发布到上面的artifacts的构建系统，比如Maven或者Gradle。

Spring Boot也可以和其他的构建系统协同工作，比如，ANT，但可能得不到更好的支持。

3.1.1 依赖管理

Spring Boot的每一个发行版都会提供一个它支持的精选依赖列表。实际上，在构建配置里你不需要提供任何依赖的版本，因为Spring Boot已经替你管理好了。当更新Spring Boot时，那些依赖也会一起更新。

3.2.1 使用“默认”的包

3.8.2 自动重启

当类路径上的文件被修改了之后，使用了spring-boot-devtools的应用程序会自动重启。这在使用IDE的时候非常有用，它会迅速的给出代码变化后的反馈。默认情况下，classpath下任何指向文件夹的实体都会被监控。但有一些资源，比如静态assets和视图模板是不需要重启应用程序的。

​ 触发重启

因为开发者工具监控了类路径下的资源，唯一的触发重启的方式就是更新类路径下的文件。不同的IDE有不同的方法触发文件更新。在Eclipse下，保存一个修改的文件，会触发重启，在IDEA中，则是构建项目。

只要启动了forking，还可以使用支持的构建插件（Maven和Gradle）来启动应用程序，因为开发者工具需要一个巨鹿的应用类加载器才能正常运行。默认情况下，Gradle和Maven插件会复制应用程序的进程。

自动重启可以和LiveReload一起很好地工作，可以参考LiveReload章节获取更多的细节。如果你使用了JRebel，自动重启将会因为类动态加载而被禁用。但其他的开发工具功能（比如，LiveReload和属性覆盖）还是可以使用的。

DevTools依赖应用上下文的shutdown钩子来关闭处于重启过程的应用，如果禁用shutdown钩子（SpringApplication.setRegisterShutdownHook(false)），它将不能正常工作。

当判断类路径下实体的改变是否应该触发重启时，开发者工具将自动忽略一下工程：spring-boot，spring-boot-devtools,spring-boot-autoconfigure，spring-boot-actuator和spring-boot-starter。

开发者工具需要定制应用上下文（ApplicationContext）需要的ResourceLoader。如果你的应用已经提供了一个，它将被包装。应用上下文（ApplicationContext)上的getResource方法是不支持被直接覆盖的。

​ Restart VS Reload

Spring Boot提供的重启技术是通过使用两个类加载器实现的。没有变化的类（比如，第三方库，jar包）会被加载进一个base(基础)类加载器。你正在开发的类将会被加载进一个重启（restart）类加载器。

当应用重启时，restart类加载器会被丢弃，然后创建一个新的。这种方法意味着应用重启通常比冷启动（cold start）要快得多，因为基础类加载器已经可用并准备就绪了。

如果你觉得对你的应用来说，重启还不够快，或者你碰到了类加载问题，你应该考虑下重载（reloading）相关技术，比如，JRebel。这些技术是通过重写它们加载过的类实现的。

在一个慢部署的环境中，静默期可能时间不足，类文件的更新可能被分为好几批。第一批类文件的更新上传之后，服务器就会重启，基于此，剩下的就不能被发送到服务器了。

这通常表现为RemoteSpringApplication日志中的一个关于一些类上传失败的warning和紧跟着的重试。但这样经常导致应用代码不一致和在第一批类更新上传之后的重启失败。如果你经常碰到这些问题，尝试着根据自己的开发环境增加spring.devtools.restart.poll-interval和spring.devtools.restart.quiet-period参数。参考Configuring File System Watcher章节以便了解如何配置这些参数。

远端应用处在运行状态时，文件才会被监控，如果你在远端应用启动之前修改了一个文件，它将不会被推送到远端服务器。

4.1.2 懒初始化

SpringApplication允许一个应用懒初始化。当懒初始化被开启之后，beans会在他们被引用的时候创建，而不是在应用启动时。这样一来，应用启动的时间将会被大幅减少。在一个Web应用中，开启懒初始化会导致许多web相关的bean不被初始化，直到一个相关的HTTP请求到达。

懒初始化的一个缺点就是它延长了问题被发现的时间。如果一个懒初始化的bean被错误地配置了，在启动的时候，这个错误将不会被发现，只会出现在这个bean被初始化的时候。还必须注意的是，JVM要有足够的内存来容纳应容程序的所有BEAN，而不仅仅是那些在初始化时候的bean。出于这些原因，默认情况下，懒初始化是不开启的。并且，在开启懒初始化之前，我们建议对JVM的堆大小进行调整。

懒初始化可以通过程序来实现。使用SpringApplication的setLazyInitialization方法或者SpringApplicationBuilder的lazyInitialization方法即可。
